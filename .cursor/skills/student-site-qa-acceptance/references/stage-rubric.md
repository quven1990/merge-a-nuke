# 上线前 QA 验收 阶段判断规则

## 本阶段目标

用真实用户任务做最终验收，输出 GO/NO-GO、P0/P1/P2、证据和返修闭环。

## 必须保留的内部流程精华（已脱敏）

### 1. 准入检查
确认 PM/SEO/合规必要 gate；没有就标风险，不盲目 GO。

### 2. 5 秒测试
首屏能否说清 What/Who/Why/CTA，核心价值是否先出现。

### 3. 真实任务
按用户任务点击，不只看页面存在；工具、地图、目录、会员、内容、后台按站点类型测。

### 4. 技术 smoke
HTTP 200/301、HTTPS、robots、sitemap、canonical、H1、metadata、schema、链接、移动端。

### 5. 权限和数据
登录/会员/支付/订单/上传/API 错误态按真实路径测；无权限时明确 BLOCKED。

### 6. 分级结论
P0 阻断上线；P1 修复后可复测；P2 可上线后跟进。

### 7. 返修闭环
修复完成后必须独立 Re-QA，不能由实现者自证。


## 质量门槛

- [ ] 不是只测首页 200
- [ ] 移动端真实点击过
- [ ] 所有 P0 有复现步骤
- [ ] GO 结论有证据支撑
- [ ] 修复项有复测记录

## 常见失败模式

- 用 DOM 存在代替真实点击
- 忽略移动端触控和横向滚动
- 把 preview 域名 sitemap 指向生产误判为 P0
- 没有测试账号却假装会员流程通过

## 脱敏原则

- 不写服务器路径、真实 token、私有账号、内部群聊或生产数据库细节。
- 保留判断标准、交接格式、质量闸门和复盘口径。
- 对学员可执行：先自然语言 Prompt，脚本只是可选。

## 2026-06 统一判分补充

本阶段无论具体产物是什么，都必须满足以下新版做站实践：

- 有明确输入来源和证据；没有证据的结论标 `待确认` 或 `missing_evidence`。
- 有下游可消费合同：路由、文案、数据、设计、接口、验收项不能只写自然语言愿望。
- 有 BLOCKED / NEEDS_REPAIR / DONE 的状态判断，以及解锁动作。
- 涉及生产、付费、公开发布、外部账号、预算、真实用户数据时，必须保留人工确认门。
- 输出必须能被总控放回 Kanban / project-control，供下一阶段继续。
